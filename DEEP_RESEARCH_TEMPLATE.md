# AI Content Share Deep Research Template

## Objective

Build and refresh a literature-review-style evidence table that helps answer:

`How much of the content on the internet is AI generated?`

This is not meant to be perfect. It is meant to be good, caveated, and reasonable.

The downstream use case is a time-series scatter chart of estimates across years. The table should preserve the spread of relevant estimates, not smooth it away too early.

## Persistent Output

The output of the deep research task should be the table itself.

- do not create a separate narrative document just to hold that table
- read the existing output table first
- read the existing refresh log first
- keep valid old rows
- append genuinely new rows
- improve old rows only when a better original source is found
- do not re-add the same estimate just because search found it again

Use a lightweight refresh log alongside the table:

- [data/ai-content-meta-review/ai_content_meta_review_refresh_log.md](</home/projects/Dead Internet Theory/data/ai-content-meta-review/ai_content_meta_review_refresh_log.md>)

The log is there to reduce repeated searching and make each refresh auditable.

## Output Table

| series | year | publication_date | value | source | notes |
|---|---:|---|---|---|---|
| Example series | 2024 | 2025-04-01 | 50% | https://example.com/source | Short summary of what was measured, how it was measured, why it matters, and the main caveat. |

Rules:

- multiple rows can exist for the same year
- that is expected
- only relevant estimates belong here
- but do collapse to one value per study per year

## Field Definitions

- `series`
  - the study or report name
  - use one row per study per year
- `year`
  - the year the estimate refers to, not just the publish date
- `publication_date`
  - inferred publication or release date for the source in `YYYY-MM-DD` form
  - use the source page date when available
  - if the source only exposes year-month, use the first of the month
  - if the date is only recoverable indirectly from the source URL or page metadata, that is acceptable
- `value`
  - the quantitative estimate itself
  - if the source gives a range, keep the range rather than forcing a fake single number
- `source`
  - the URL of the best original source available
- `notes`
  - 3-4 sentences max
  - explain what is being measured, how it was measured, why it is relevant, and the main caveat

If one source gives multiple years, create one row per year. Repeat the same `source` URL, `publication_date`, and `series` label across those rows.

## Core Filter

Before including any source, ask:

`Does this help answer how much of the content on the internet is AI generated?`

If the answer is not clearly yes, or at least yes as a close proxy, leave it out.

This is the most important filter in the whole process.

Close enough examples:

- estimate of AI-generated share of web articles
- estimate of synthetic share in a large online content corpus
- estimate of AI-written share on a major publishing platform

Not close enough:

- AI bot traffic share
- AI search usage
- chatbot visits
- bot attacks
- referral traffic
- model training traffic

## Source Priority

1. Academic papers and research reports first
2. Within those, heavily prefer sources that include time series themselves
3. Reputable news second, only if they contain a usable quantitative estimate
4. Do not use weak blogs, clickbait, or recycled summaries

When using news:

- prefer major publications with clear attribution
- only keep the article if the number can be traced to a study, report, dataset, or credible company claim
- if the original source can be found, use that instead

## Search Strategy

Run 2-3 search passes with slightly different queries.

High-level approach:

1. start broad and gather the strongest obvious sources first
2. build the table from those sources
3. identify which years are still thin
4. run targeted gap-filling searches for those weaker years
5. only conclude data is sparse after multiple search styles fail

Suggested queries:

- `AI generated content share internet study`
- `AI generated web content percentage`
- `synthetic content online estimate`
- `AI written articles percentage study`
- `AI generated content Common Crawl study`
- `share of web content generated by AI`
- `AI generated content research report`

Also search by year when refreshing:

- `2026 AI generated content estimate`
- `2026 AI written web content study`

Search order:

- Google Scholar
- arXiv
- direct web search for research reports
- reputable news only after the academic / report pass

Before starting a refresh:

1. read the existing table
2. read the refresh log
3. note which years, domains, and query families still look thin

Prefer sources that already provide:

- multi-year estimates
- repeated method over time
- a table of values
- or a chart readable year by year

## Selection Rules

For each candidate source:

1. Check that it contains a real number
2. Check that the number refers to content, not traffic or usage
3. Check what population it refers to
4. Check whether it is close enough to the core question
5. Check whether the source is original
6. Check whether it contains a time series or only a single-year estimate
7. Decide whether it is strong enough to include

If any of those steps fail, leave it out.

Decision framework:

- include it when it is relevant enough to the real question
- that means it should estimate the share of AI-generated content within a clearly defined content set
- the content set can be broad or narrow, as long as it is still genuinely content
- exclude it when it becomes too adjacent, too indirect, or too far into traffic, usage, or behavior

Good examples:

- share of new web articles that are AI-generated
- share of Wikipedia pages that are AI-generated
- share of social-media posts that are AI-generated
- share of scientific articles that are AI-generated

Borderline examples:

- AI-assisted writing estimates can be included if they are close enough to a content-share estimate and the notes make that clear
- heuristic or detector-based estimates can be included if they are still trying to estimate AI-generated share within a content set

Bad examples:

- bot traffic share
- AI referral traffic
- AI usage rates
- attack or abuse metrics

## Study-Year Rule

Use one value per study per year.

That means:

- if a study has a time series, keep one row per year
- if a study gives a range for a year, use the midpoint
- if a study gives one broad overall estimate and several narrower subtype estimates for the same year, keep the broad overall estimate
- if a study only gives multiple equally valid sub-estimates for the same year, collapse them to one study-year value using the median

The goal is to stop one study from effectively casting multiple votes in the same year.

## Exclusion Rules

Reject:

- vague claims with no number
- opinion pieces or speculation
- stats about bots, traffic, referrals, or AI usage
- metrics too far from the core question

Example exclusion:

- `AI bot traffic makes up 55% of internet usage`
  - not relevant here because it measures traffic or automation, not AI-generated content

More exclusion examples:

- `% of users using ChatGPT`
- `% of search referrals coming from AI`
- `% of web traffic from bots`
- `% of attacks that are AI-enabled`

These may be useful elsewhere in the dashboard, but not in this table.

## Deduplication Rules

Use one row per distinct estimate.

If multiple sources point to the same estimate:

- keep the original source if possible
- otherwise keep the strongest secondary source
- do not repeat the same number across multiple rows just because many articles mention it

Treat as duplicates when they are clearly the same:

- same number
- same source
- same year
- same underlying study or report

Do not treat as duplicates just because they share a year. Keep them separate if they differ in:

- methodology
- scope
- population
- reported value

Repeated source URLs across different years in the same `series` are expected and should not be deduplicated away.

Deduplicate in two places:

1. within the new search results
2. against the existing table

The goal is:

- append new information
- not re-append old information
- preserve real disagreement between relevant estimates

## Refresh Log

Each refresh should update the lightweight log as well as the table.

For each run, record:

- date
- short scope summary
- strongest sources added
- notable dead ends or rejected result classes
- years or domains that still look thin

This does not need to be long. The point is to avoid repeating the same search patterns blindly next time.

## Notes Guidance

The `notes` field should be short and tooltip-ready.

Write it as exactly three short sentences:

1. claim: what the source found, including the number and scope
2. method: how it was measured
3. caveat: the main limitation or reason to be cautious

Do not add a separate relevance sentence. The tooltip now renders `notes` in this 3-sentence structure directly.

Keep it short. Do not turn it into a full abstract.

Example:

`The source estimated that 35% of newly published English-language web articles were AI-generated. It sampled pages from Common Crawl and classified them with an AI detector. The estimate is narrower than all internet content and depends on detector quality.`

## Time-Series Extraction Rule

If a source contains a time series:

- extract one row per year
- use the same `series` label for all rows from that source series
- repeat the source URL across those rows
- keep the values separate rather than averaging them

If the source provides only a chart:

- read the values off the chart as carefully as possible
- say clearly in `notes` that the values were chart-read
- only do this when the chart is readable enough to be reasonably accurate

## Quality Bar

The goal is not perfection.

The goal is that a university professor might have issues with parts of the review, but would still accept that it is a reasonable and caveated summary of the available evidence.

Prefer fewer strong estimates over many weak ones.

## Monthly Refresh Process

Each refresh should:

1. read the existing table first
2. read the existing refresh log
3. search for new academic papers and research reports
4. search reputable news for new quantified estimates
5. filter candidates using the core question
6. deduplicate the new results against each other
7. deduplicate against the existing table
8. append only genuinely new rows
9. update old rows only if a better original source is found
10. update the refresh log with what was added, what failed, and what still looks thin
11. leave gaps where no credible estimate exists

After updating, sort the table by:

1. year
2. series
3. value

Do not:

- overwrite the table from scratch
- duplicate rows because the same source appears again
- delete older valid rows just because no new replacement was found

## Output Expectations

Return:

- the updated table
- a deduplicated source list
- a 1-2 sentence trend summary

The table should be sorted by year so it can be plotted directly as a time-series scatter.

The trend summary should not overclaim.

Example:

`The evidence remains sparse, but the published estimates that do exist generally point toward a rising share of AI-generated content online. Most strong estimates are still narrow in scope, often focusing on new web articles or specific platforms rather than the whole internet.`

## Desired Outcome

Over time, this should become a sparse but credible evidence table showing how published estimates and close proxies evolve, without pretending there is one perfect authoritative number.
